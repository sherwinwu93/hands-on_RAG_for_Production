## Challenges with RAG in Production
### Response Quality and Reduced Hallucinations回答质量 减少幻觉
回答质量差的原因:
1. 没有相关的数据
   1. 例子: 1. 根据现有资料完全无法回答; 2. 根据现有资料只能回答一半
   2. 解决方法: POC: 找到缺失文档直接喂; Production: 先遇到stage,在stage里面测试,测试通过,再搞到生产
   3. 困难: 只有专家才能判断文档是否相关,是否过时
2. 检索管道弱: 生产文件多了,检索效果会变差; 需要更好的filtering,reranking,search(store)
3. LLM hallucinations: 检索到数据有部分数据,又没完全包含问题, 会导致LLM自己补充,造成幻觉.
4. Prompt Engineering: prompt里面明确告知LLM如果不能根据上下文回答,就说自己不知道.(给LLM明确的步骤)

### High Latency高延迟
retrieval components: 不超过300ms
LLM: 2-3s,或者5-10s(推理模型)

生产造成延迟的原因:
- 向量库要正确索引
- 混合搜索要处理更多数据
- reranker要处理更多candidates
- LLM要处理更多context
- 主要: 不单单要保持平均延迟,还要维持95%的延迟

处理方法:
- 并发+自动扩容
- 多LLMs
- 软硬件加速
- 高效数据索引
- 缓存

#### Parallelization and auto-scaling
设计模式: 一套解耦的微服务+一个无状态的编制服务
- 编制服务: 接收和分发用户请求
- 微服务: 一个服务查向量,一个服务查关键字(同时)
- 设计的关键,是分组,并且可以根据其性能瓶颈扩容(CPU,GPU,数据库连接)

#### Caching
可以把缓存应用在:
- Full response cache: 缓存问题,直接从缓存中返回答案
   缓存语义比缓存问题本身更好
- Retrieval cache: 缓存向量库/关键字结果
- Chunk cache: 根据向量库拿到chunk_id后,直接从chunk缓存拿到chunk

缓存成功的要点在于缓存要及时更新,订阅更新缓存比单纯过时更新要好很多
- Cache invalidation: 缓存订阅更新
- Eviction strategy: 缓存过期LRU(least recently used)
- Horizontal scaling: redis集群

### Data Security and Privacy数据安全,隐私
三个层面的安全:
- ingestion层
- 向量库和关键字数据库
- 生成层
#### Ingestion layer security ingestion层安全
用标准加密协议来传输数据
redaction strategy:
- masking: 敏感数据用*代替. 
  这里有更好的方式,既加密又不影响语义: eg. [DOCTOR_NAME] prescribed [MEDICATION] to [PATIENT_NAME]
- nulling: 移除敏感数据

#### Data store safeguards 数据存储安全
向量库,关键字数据库,图库的核心要求都是:
- 加密
- role-based access control: 按角色分配权限
- 不存储明文密码
